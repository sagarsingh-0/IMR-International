Abstract

The proliferation of web-based applications and digital platforms has created an unprecedented demand for user behavior analytics, enabling organizations to make data-driven decisions about user experience, content strategy, and service optimization. However, conventional web analytics systems operate through pervasive data collection mechanisms that routinely capture, store, and process personally identifiable information without adequate transparency or user consent, raising serious ethical, legal, and privacy concerns. Regulations such as the General Data Protection Regulation (GDPR) in Europe and the California Consumer Privacy Act (CCPA) in the United States have established stringent legal frameworks requiring explicit user consent before any data collection activity, mandating that organizations rethink their approach to user behavior monitoring.

This project presents the design, development, and full-stack implementation of a Consent-Based Anonymous User Behavior Monitoring System — a privacy-first web analytics platform integrated into the official website of IMR International (International Institute for Management and Research), Bhubaneswar, Odisha. The system is built upon a modern technology stack comprising React with Vite for the frontend, Node.js with Express for the backend API server, and PostgreSQL with Drizzle ORM for persistent data storage.

The core innovation of this system lies in its strict adherence to the principle of informed, explicit, and revocable consent. The system presents users with a clearly worded consent banner upon their first visit to the website. Data collection through detailed behavioral analytics begins strictly and exclusively when users grant affirmative consent. Users who decline or withdraw consent are not tracked beyond a minimal server-side visit log that records no personal identifiers. All session identifiers are generated using the Web Cryptography API and immediately hashed using the SHA-256 cryptographic algorithm before any storage, ensuring that no raw personal data ever touches the database.

The system collects and analyzes several categories of behavioral signals including page view sequences, navigation patterns, user engagement duration, click interactions, and session characteristics. These signals are aggregated and visualized through a comprehensive admin analytics dashboard featuring interactive charts rendered with the Recharts library, tabular engagement reports, and exportable CSV data snapshots. A complementary database viewer interface allows authorized administrators to inspect all database tables in real time, with all data access enforced through session-based authentication.

Additionally, the system incorporates a server-side visitor log that records minimal, non-identifiable information (page visited, language preference, hashed IP fragment, and timestamp) for every visit regardless of consent status, providing basic operational metrics without compromising user privacy. This two-tier approach — consent-gated detailed analytics and consent-free minimal logging — represents a balanced implementation of privacy-by-design principles.

The project demonstrates that rich, actionable behavioral insights can be obtained while fully respecting user privacy and maintaining compliance with international data protection regulations. The resulting system serves as both a functional production deployment for IMR International and a reference implementation of ethical, consent-aware web analytics architecture.

Introduction

1.1 Background and Motivation

The World Wide Web, since its commercialization in the mid-1990s, has grown into the primary medium through which individuals, institutions, and businesses communicate, transact, and consume information. As of 2025, over 5.5 billion people access the internet globally, generating extraordinary volumes of behavioral data through every click, scroll, navigation, and search query they perform. This behavioral data has become one of the most valuable commodities in the digital economy, enabling organizations to optimize user experiences, personalize content, forecast demand, and allocate resources with precision previously unimaginable.

Web analytics — the systematic collection, measurement, analysis, and reporting of website usage data — has been a cornerstone of online strategy since the late 1990s. The earliest analytics systems were rudimentary log file analyzers that parsed server access logs to count unique visitors and page views. The launch of sophisticated client-side tracking platforms, most notably Google Analytics in 2005, fundamentally transformed the discipline, enabling organizations to track detailed user journeys, funnel drop-offs, geographic distributions, device breakdowns, and conversion rates through lightweight JavaScript snippets embedded in web pages.

However, this revolution in analytical capability came at a profound cost to user privacy. Traditional web analytics platforms routinely assign persistent cookies or device fingerprints to each user, enabling the construction of detailed behavioral profiles that persist across sessions, devices, and even websites through third-party tracking networks. Users are often entirely unaware that their browsing behavior is being monitored, correlated, and sold to data brokers and advertising platforms. The business model of dominant analytics providers has historically depended on this broad, pervasive surveillance of user behavior, treating user data as a product rather than a resource held in trust.

The ethical dimensions of this data collection model have been a subject of growing public, academic, and legislative concern since at least the early 2010s. A series of high-profile data breaches, journalistic investigations, and academic studies revealed the extent to which personal data collected ostensibly for analytics purposes was being repurposed for commercial profiling, political targeting, and discriminatory algorithmic decision-making. The Cambridge Analytica scandal of 2018, in which Facebook user data was harvested at scale and used for political manipulation, brought these concerns to global mainstream attention and catalyzed unprecedented legislative action.

The European Union's General Data Protection Regulation (GDPR), which came into full effect on 25 May 2018, represents the most comprehensive legislative response to date. GDPR establishes that personal data may only be collected and processed with a clear, specific, informed, and freely given legal basis — with explicit consent being one of the six recognized legal bases. It grants individuals extensive rights over their personal data including the right to access, rectification, erasure ("right to be forgotten"), and portability. Non-compliance carries substantial financial penalties of up to €20 million or 4% of global annual turnover, whichever is greater. GDPR has had a profound effect on how organizations worldwide approach data collection, even those not based in the European Union, due to its extraterritorial scope covering any processing of EU residents' personal data.

In the United States, the California Consumer Privacy Act (CCPA) of 2018 and its subsequent amendment, the California Privacy Rights Act (CPRA) of 2020, established similar rights for California residents, with additional state-level regulations emerging in Virginia, Colorado, Connecticut, Utah, and other jurisdictions. India's Digital Personal Data Protection Act (DPDPA) of 2023 has introduced similar obligations for organizations processing personal data of Indian citizens. This global legislative trend signals a fundamental paradigm shift: user data collection must be transparent, purposeful, and consensual.

Within this shifting landscape, higher educational institutions face a particular challenge. Universities and colleges operate websites that serve multiple constituencies — prospective students conducting program research, current students accessing academic resources, faculty seeking institutional information, and parents evaluating educational options. These visitors have legitimate privacy expectations, particularly when engaging with an educational institution that ostensibly upholds academic integrity and ethical conduct. The tension between the operational need for website analytics and the ethical obligation to respect visitor privacy presents a design challenge that this project directly addresses.

IMR International (International Institute for Management and Research), located in Bhubaneswar, Odisha, operates a multi-page institutional website serving thousands of visitors monthly. The institute offers a range of undergraduate and postgraduate programs in management, computer applications, and data science, along with numerous professional certification programs. Like any modern educational institution, IMR International has a legitimate interest in understanding how visitors interact with its website — which programs attract the most interest, how prospective students navigate the admission process, what content drives enquiry form submissions, and how overall engagement trends over time. These insights directly inform academic marketing strategy, curriculum communication, and digital outreach efforts.

Prior to this project, no systematic, privacy-compliant user behavior monitoring system existed for the IMR International website. The institute either relied on no analytics at all — losing valuable insights — or would have needed to deploy conventional analytics platforms that would collect data without proper consent mechanisms. This gap identified both the practical need and the academic opportunity that motivated this project.

1.2 Problem Statement

The fundamental problem addressed by this project can be articulated at two distinct levels: a technical problem and an ethical-regulatory problem. Both dimensions are inextricably linked, and any solution must address them simultaneously to be considered valid and complete.

At the technical level, the problem is the absence of a functional, lightweight, self-hosted user behavior monitoring system that can be integrated into an institutional website without dependence on third-party analytics platforms. Third-party platforms like Google Analytics, while feature-rich, present several technical disadvantages for institutional deployments: they send all user data to external servers controlled by the platform vendor, they are frequently blocked by browser privacy extensions and ad-blockers, they impose data retention limits and vendor lock-in, and they provide limited customization for institution-specific analytics needs. A self-hosted solution eliminates these dependencies while giving the institution complete ownership and control over its analytics data.

At the ethical and regulatory level, the problem is that conventional tracking systems — whether third-party or self-hosted — routinely violate the principle of informed consent by initiating data collection before users have been informed about it and provided affirmative approval. This practice is now explicitly illegal under GDPR and several other privacy regulations for websites serving users in regulated jurisdictions. Educational institutions, which have particular obligations around ethical conduct, cannot in good conscience operate tracking systems that disregard their users' right to informational self-determination.

More specifically, the following concrete problems are identified and addressed by this project:

  • Absence of consent gateway: Existing web analytics deployments typically initiate tracking scripts immediately on page load, before any interaction with the user about data collection. This violates the GDPR requirement that consent be obtained before processing begins and that consent be freely given rather than assumed.
  • Storage of personally identifiable information: Standard analytics platforms assign persistent, unique identifiers (typically via HTTP cookies or localStorage) that allow tracking of individual users across sessions. This identifier, combined with behavioral data, constitutes personal data under GDPR even if no name or email address is associated with it, because it enables singling out individuals.
  • Lack of data transparency: Users visiting institutional websites typically have no visibility into what data is being collected about them, by whom, how long it is retained, or for what purposes it will be used. This opacity violates the transparency principle that is foundational to all modern privacy frameworks.
  • Irrevocability of tracking: Once tracking begins under traditional systems, it continues indefinitely unless the user takes active steps to clear cookies or enable privacy protection measures — steps that most non-technical users are unaware of or unable to perform. Consent, to be valid under GDPR, must be as easy to withdraw as to grant.
  • No administrative insights interface: Even institutions that care about user privacy often lack a practical interface through which authorized personnel can review aggregated behavioral data, generate reports, and export information for decision-making without accessing raw personal records.

1.3 Project Overview

This project develops, implements, and deploys a full-stack web application system — the Consent-Based Anonymous User Behavior Monitoring System — integrated into the official website of IMR International, Bhubaneswar. The system operates through a carefully designed pipeline of user interaction, consent management, anonymous data collection, server-side processing, persistent storage, and administrative visualization.

The implemented solution consists of three interconnected layers. The presentation layer is a React-based single-page application built with Vite, comprising a complete institutional website with multiple content pages covering academic programs, certifications, focus centers, admission information, and institutional background. The application layer is an Express.js API server running on Node.js, responsible for processing all data collection events, enforcing authentication for administrative access, and serving aggregated analytics data. The data layer is a PostgreSQL relational database managed through the Drizzle ORM, storing behavioral events, consent decisions, visitor records, user registrations, enquiry submissions, and chatbot interactions in a structured, queryable format.

The consent management subsystem is the most critical component of the architecture. Upon a visitor's first interaction with the website, a prominently displayed consent banner appears, clearly explaining that the website uses anonymized behavioral tracking and asking for the visitor's explicit approval or refusal. The banner cannot be dismissed without making an explicit choice, and the user's decision is stored in the browser's localStorage for subsequent visits. Detailed behavioral tracking — recording page view sequences, navigation patterns, and click interactions — begins only when the user has granted consent. Users who decline are served the same website content but contribute only to a minimal server-side visitor count with no behavioral detail recorded.

Session identification, when tracking is active, uses the browser's built-in Web Cryptography API (crypto.randomUUID()) to generate a tab-scoped random identifier that exists only for the duration of the browser session. This identifier is immediately processed through the SHA-256 hash function before transmission to the server, meaning that the server never stores or processes the raw session identifier. The resulting hash is deterministic within a session (enabling session-level aggregation) but computationally irreversible, making re-identification of individual users mathematically infeasible.

The administrative interface provides authorized personnel with a comprehensive analytics dashboard featuring time-series visualizations of traffic patterns, page popularity rankings, user engagement metrics, consent rate tracking, and device/language distribution analysis. A complementary database viewer allows inspection of all system tables with full-text search and CSV export capabilities. Access to all administrative functions is protected by session-based authentication with secure credentials.

1.5.2 Limitations

The following aspects fall outside the scope of the current implementation and represent acknowledged limitations:

  • Cross-device tracking: The system explicitly does not attempt to correlate sessions across different devices or browsers used by the same individual, as this would require storing user identifiers that approach personally identifiable information.
  • Real-time push updates: The admin dashboard refreshes data on demand rather than through real-time WebSocket push updates. High-frequency real-time tracking is beyond the current scope.

  • Mobile application tracking: The system is designed specifically for web browser environments and does not extend to native mobile application analytics.
  • Automated consent withdrawal mechanisms: While users can withdraw consent by declining on revisit, no automated "forget me" deletion workflow is implemented in the current version.
  • Multi-language consent banner: The current consent UI is implemented in English only. Localization for Odia and other regional languages is identified as a future enhancement.

Literature Review

2.1 Evolution of Web Analytics

The practice of measuring web traffic has existed almost as long as the World Wide Web itself. The earliest form of web analytics emerged in the early 1990s with the analysis of server log files — plain text records generated by web server software that captured each request for a resource (HTML page, image, stylesheet) along with the requesting IP address, timestamp, and HTTP status code. Software tools such as Analog (released 1995) and AWStats (2000) parsed these log files and produced statistical summaries of page requests, unique IP addresses, referring URLs, and browser types. While technically functional, log-based analytics suffered from several fundamental limitations: IP addresses are shared by multiple users on networks using Network Address Translation (NAT), crawlers and bots inflate page request counts, and there is no mechanism to distinguish a returning visitor from a new one with the same IP address.

The introduction of JavaScript-based client-side tracking in the late 1990s addressed many of these limitations. By embedding a JavaScript snippet in web pages, analytics systems could set persistent HTTP cookies in the visitor's browser — small text files that the browser would transmit with every subsequent request to the same domain. This enabled the identification and distinction of unique visitors from returning visitors, the measurement of visit duration (previously impossible with log-based systems), and the tracking of multi-page sessions as coherent user journeys. Urchin Software Corporation, founded in 1998, was a pioneer in this approach, developing sophisticated web analytics software that could track complete visitor sessions.

Google's acquisition of Urchin in April 2005 and the subsequent release of Google Analytics as a free service in November 2005 fundamentally democratized web analytics. Google Analytics provided unprecedented analytical depth — visitor geography, browser and device breakdown, traffic source attribution, goal conversion tracking, and eventually e-commerce reporting — at no direct monetary cost to website operators. The business model was predicated on Google gaining access to the browsing data of billions of users across millions of websites, data that fed its advertising targeting capabilities. Within a decade, Google Analytics was deployed on over 50 million websites globally, making it the most widely deployed JavaScript application in history.

The period from 2010 to 2020 saw several important developments in web analytics. The rise of mobile computing necessitated cross-device tracking capabilities, pushing analytics platforms to develop probabilistic matching algorithms and authenticated user tracking mechanisms. The growth of single-page application (SPA) frameworks like React, Angular, and Vue.js challenged traditional page-view based measurement paradigms, requiring analytics systems to adapt to virtual navigation events rather than actual HTTP page loads. The widespread adoption of HTTPS encrypted connections by 2015 eliminated the ability of intermediary parties to inspect web traffic, concentrating analytics power in client-side JavaScript and server-side first-party systems.

The late 2010s and early 2020s brought a countermovement against pervasive tracking. Apple's Intelligent Tracking Prevention (ITP) in Safari (2017) and Firefox's Enhanced Tracking Protection (ETP) began systematically blocking third-party tracking cookies. Google's announced (and subsequently delayed) deprecation of third-party cookies in Chrome signaled the eventual end of the cross-site tracking model that underpinned much of the digital advertising economy. Browser extensions like uBlock Origin and Privacy Badger, used by an estimated 30–40% of desktop users, actively blocked analytics scripts including Google Analytics, creating systematic undercounting in conventional analytics systems. These developments accelerated interest in privacy-preserving, consent-based, and cookieless analytics alternatives.

A new generation of privacy-focused analytics platforms emerged to address this gap, including Plausible Analytics, Fathom Analytics, Matomo (open source), and Simple Analytics. These platforms share a common design philosophy: measuring meaningful traffic signals without setting tracking cookies, without using third-party data sharing, without storing IP addresses, and in most cases without requiring user consent under GDPR because they are designed to operate without processing personal data. The present project is positioned within this movement but extends it through a self-hosted, purpose-built implementation with explicit consent management and institutional-specific analytics capabilities.

2.2 Privacy Regulations and Legal Framework

The legal landscape governing web-based data collection has undergone rapid and significant transformation since 2016. Understanding this landscape is essential for appreciating both the motivation for consent-based analytics and the design decisions made in this project.

The European Union's General Data Protection Regulation (GDPR), published in the Official Journal of the European Union on 4 May 2016 and entering into application on 25 May 2018, represents the most comprehensive and influential data protection legislation yet enacted. GDPR applies to any organization that processes the personal data of individuals located in the European Union, regardless of where the organization itself is based — a principle of extraterritorial applicability that has global implications. The regulation defines personal data broadly as "any information relating to an identified or identifiable natural person," and explicitly clarifies that online identifiers including IP addresses and device identifiers qualify as personal data when they can, even indirectly, lead to the identification of an individual.

GDPR establishes six lawful bases for processing personal data: consent, contract performance, legal obligation, vital interests, public task, and legitimate interests. For the specific purpose of behavioral analytics — tracking users' browsing behavior to understand engagement patterns — legitimate interests is sometimes invoked, but this basis requires a balancing test demonstrating that the controller's interests outweigh the individual's privacy interests, and is frequently challenged by data protection authorities when applied to tracking. Consent, therefore, remains the most appropriate legal basis for user behavior analytics, provided it meets GDPR's stringent standards: it must be freely given, specific, informed, and unambiguous, manifested through a clear affirmative action.

The Planet49 ruling by the Court of Justice of the European Union in October 2019 clarified that pre-ticked checkboxes do not constitute valid consent, establishing that affirmative action is essential. The subsequent Planet49 implications have been extensively litigated across EU member states, with national data protection authorities issuing guidance reinforcing that simply continuing to browse a website does not constitute consent to tracking, and that consent must be as granular, specific, and revocable as the data collection activities require.

India's legal framework for data protection has also undergone significant development. The Information Technology Act 2000 and its subsequent amendments provided an early, limited framework for digital privacy, but it was widely recognized as inadequate for contemporary data processing realities. The Supreme Court of India's landmark ruling in Justice K.S. Puttaswamy vs. Union of India (2017) recognized the right to privacy as a fundamental right under the Indian Constitution, establishing a constitutional foundation for subsequent legislative action. The Digital Personal Data Protection Act 2023 (DPDPA), signed into law on 11 August 2023, establishes consent as the primary legal basis for data processing and creates the Digital Data Protection Board as a regulatory enforcement authority. DPDPA is particularly relevant to this project as IMR International is an Indian institution primarily serving Indian users.

At the institutional level, universities and colleges in India are subject to various regulatory bodies including the University Grants Commission (UGC), the All India Council for Technical Education (AICTE), the National Board of Accreditation (NBA), and the National Assessment and Accreditation Council (NAAC). These bodies increasingly emphasize ethical conduct and governance standards that encompass digital operations, creating institutional motivation for compliance with privacy norms even where direct legal enforcement is not yet rigorous.

The legal and regulatory context described above directly informed the architectural decisions of this project. The mandatory consent banner before any tracking, the two-tier logging architecture (consent-gated detailed analytics versus consent-free minimal logging), the cryptographic anonymization of session identifiers, the prohibition on storing IP addresses in any personally identifiable form, and the data immutability policy preventing post-collection deletion or modification — all of these design choices were motivated by and are consistent with the requirements of GDPR, DPDPA, and associated regulatory guidance.

2.3 Consent Management Platforms

Consent Management Platforms (CMPs) are specialized software systems designed to manage the collection, storage, and enforcement of user consent for data processing activities in compliance with privacy regulations. The Interactive Advertising Bureau (IAB) developed the Transparency and Consent Framework (TCF) as an industry standard for CMPs, enabling interoperability between publishers, CMPs, and downstream technology vendors. Commercial CMPs such as OneTrust, Cookiebot, TrustArc, and Usercentrics provide enterprise-grade consent management solutions with features including geolocation-based consent requirement detection, multi-jurisdictional compliance coverage, granular consent categories (functional, analytics, marketing), consent string encoding for vendor interoperability, and detailed consent audit logs.

Existing academic research on consent management has identified several important patterns in user behavior regarding consent decisions. A 2019 study by Utz et al. ("(Un)informed Consent: Studying GDPR Consent Notices in the Field") found that when given a genuine choice, approximately 75–90% of users decline tracking when the consent interface is designed to present options neutrally, compared to much higher acceptance rates when consent interfaces are designed with "dark patterns" — manipulative interface designs that nudge users toward acceptance through asymmetric button prominence, pre-selected options, or excessive friction for refusal.

The design of the consent banner in this project was directly informed by these research findings. The consent banner presents acceptance and rejection options with equal visual prominence — both presented as clearly labeled buttons without color bias toward acceptance. The banner appears prominently after a brief delay to ensure the user has registered its presence, rather than as an easily overlooked footnote. The underlying philosophy is that genuinely valid consent must be genuinely informed and genuinely voluntary, and that analytics systems built on manipulated consent are not only legally questionable but ethically indefensible.

A distinct design decision in this project is the choice to implement consent management natively rather than deploying a commercial CMP. Commercial CMP scripts themselves frequently engage in data processing by transmitting consent signals to third-party networks — a somewhat ironic situation where the system managing privacy consent itself involves third-party data flows. The native implementation in this project ensures that consent management is entirely self-contained, with consent decisions stored exclusively in the user's browser localStorage and the project's own PostgreSQL database, with no data transmitted to external systems.

2.4 Anonymous Tracking Techniques

The academic and industry literature describes several technical approaches to user behavior monitoring that minimize or eliminate the collection of personally identifiable information. This section reviews the most relevant techniques and their applicability to the present system.

Cookieless Tracking: The most straightforward approach to privacy-preserving analytics is to avoid setting HTTP cookies or using persistent localStorage identifiers entirely, instead deriving session identity from available signals without writing persistent identifiers. Plausible Analytics, for example, generates daily-rotating salted hashes of IP address, user agent, and hostname combinations to count unique visitors without cookies. This approach has the advantage of GDPR compliance in many interpretations (because no persistent identifier is stored on the device), but the disadvantage of inaccuracy — the same user visiting twice in one day on the same device may or may not be counted as the same visitor depending on network conditions. The present project adopts a hybrid approach: a session-scoped UUID is generated client-side but immediately hashed before transmission, providing within-session continuity while preventing cross-session tracking.

Differential Privacy: Differential privacy, formalized by Cynthia Dwork in 2006, provides mathematical guarantees that the presence or absence of any individual's data cannot be inferred from aggregate query results by adding calibrated statistical noise to outputs. While powerful for large-scale datasets, differential privacy introduces practical complexity and requires careful parameter tuning (the privacy budget ε) that may reduce the utility of analytics for small-traffic websites like institutional sites. The present system does not implement formal differential privacy but achieves analogous protection through aggregation — all dashboard visualizations present data in aggregate form with no individual-level breakdown.

k-Anonymity: k-Anonymity, introduced by Latanya Sweeney in 1998, requires that any record in a published dataset is indistinguishable from at least k-1 other records with respect to quasi-identifying attributes. In the web analytics context, this would require suppressing or generalizing records for page paths or time windows with fewer than k visitors. While not explicitly implemented as a formal k-anonymity guarantee, the present system's aggregation approach and the exclusion of highly granular attributes (precise timestamps, specific referring URLs at the individual level) achieves practical k-anonymity for the aggregated analytics data exposed in the dashboard.

IP Anonymization: Rather than storing full IP addresses, the present system performs a one-way cryptographic hash (SHA-256) of the raw IP address before storage, retaining only the first 16 hexadecimal characters of the resulting hash. This approach, consistent with the IP anonymization features offered by Google Analytics 4, renders re-identification from the stored hash computationally infeasible while still providing a stable fingerprint within a session for deduplication purposes. IP anonymization is explicitly recognized by several national data protection authorities, including Germany's DSK (Conference of Independent Federal and State Data Protection Supervisory Authorities), as a sufficient anonymization technique rendering the resulting data no longer personal data under GDPR.

2.5 Cryptographic Privacy Techniques

The cryptographic techniques employed in this project — specifically UUID generation and SHA-256 hashing — require examination in the context of their privacy implications and security properties.

The UUID (Universally Unique Identifier) standard, defined in RFC 4122, specifies formats for 128-bit identifiers with extremely low collision probability. UUID version 4 (UUIDv4), used in this project through the browser's crypto.randomUUID() method, generates identifiers through a cryptographically secure pseudorandom number generator (CSPRNG), making the identifiers statistically unpredictable and resistant to sequential enumeration attacks. The use of the Web Cryptography API for UUID generation, rather than Math.random()-based alternatives, is significant: Math.random() is not cryptographically secure and can be predicted by adversaries, whereas CSPRNG-based UUIDs are suitable for security-sensitive contexts.

SHA-256 (Secure Hash Algorithm 256-bit) is a member of the SHA-2 family of cryptographic hash functions, standardized by the National Institute of Standards and Technology (NIST) in 2001 and widely used in security-critical applications including TLS certificate signing, digital signatures, and blockchain consensus. SHA-256 has several properties essential to its role in this system: it is deterministic (the same input always produces the same output, enabling within-session aggregation), it is one-way (computationally infeasible to recover the input from the output — the current best attacks on SHA-256 remain theoretical), and it produces a fixed-length 256-bit (32-byte, 64-hexadecimal character) digest regardless of input size. As of 2025, SHA-256 has no known practical vulnerabilities and is considered appropriate for privacy-sensitive applications of this nature.

bcrypt, the algorithm used for password hashing in the user authentication module of this project, is specifically designed for password storage and differs from SHA-256 in important ways. Unlike SHA-256, bcrypt incorporates a work factor (cost parameter) that controls computational expense, making brute-force and rainbow table attacks progressively more difficult as hardware improves. The bcrypt implementation used (the bcryptjs Node.js library) applies a work factor of 12, requiring approximately 250 milliseconds of computation per hash on modern hardware — acceptable for login operations but prohibitive for bulk brute-force attacks. bcrypt also incorporates a random salt to prevent rainbow table attacks against common passwords.

2.6 Review of Existing Systems

To properly position the contribution of this project, it is necessary to critically examine existing web analytics systems with respect to privacy compliance, consent management, data ownership, and analytical capability.

Google Analytics 4 (GA4): Released in 2020 as the successor to Universal Analytics, GA4 represents Google's adaptation to the post-cookie, privacy-conscious environment. GA4 introduces several privacy enhancements including optional IP anonymization, consent mode integration (which adjusts measurement when consent is declined), and behavioral modeling to estimate metrics for non-consenting users using machine learning. However, GA4 remains fundamentally a third-party analytics service that transmits all user data to Google's servers, requires non-trivial configuration to achieve GDPR compliance, and has been ruled non-compliant with GDPR by several EU data protection authorities (including Austria's DSB and France's CNIL) due to the transfer of EU user data to US servers under the pre-Schrems II legal framework.

Matomo (formerly Piwik): Matomo is an open-source web analytics platform that can be self-hosted, addressing the data sovereignty concern of third-party services. Matomo offers a comprehensive feature set comparable to Google Analytics, including page heatmaps, session recording, funnel analysis, and A/B testing. It provides a configurable consent management mode that can anonymize data collection. However, Matomo's full feature set is complex to deploy and maintain for non-technical users, and its self-hosted version requires server administration expertise. The cloud-hosted Matomo option reintroduces third-party data dependencies. Matomo does not implement cryptographic session anonymization natively — it relies on cookie-based tracking by default even in its privacy mode.

Plausible Analytics: Plausible is a privacy-focused, cookieless analytics platform that deliberately excludes personal data from its data model. It records page views, referrer sources, browser types, device categories, and geographic regions without using cookies, session IDs, or persistent identifiers. By design, it claims to be GDPR-compliant without requiring consent banners. However, Plausible is a third-party SaaS service that requires subscription fees, does not provide behavioral tracking at the session level, and offers no mechanism for collecting custom events or enrollment-specific institutional metrics.

Fathom Analytics: Similar to Plausible in its privacy-first design, Fathom is another subscription SaaS analytics service with cookieless, consent-free tracking. It offers real-time traffic visibility, goal conversion tracking, and a clean dashboard interface. Like Plausible, it lacks session-level behavioral tracking and institutional customization capabilities.

2.7 Research Gap and Contribution

The literature review reveals a clear research gap: while both privacy-preserving analytics platforms and consent management systems exist independently, there is limited work demonstrating their tight integration in a self-hosted, purpose-built, full-stack educational institution context with cryptographic session anonymization. 3.4 Use Case Analysis

Use case analysis identifies the distinct actors that interact with the system and specifies the concrete interaction sequences that constitute meaningful use of the system's functionality. Three primary actors are identified: the Anonymous Visitor (a user who has not granted tracking consent or has not yet encountered the consent banner), the Consenting Visitor (a user who has granted tracking consent), and the Administrator (an authenticated staff member with access to the analytics and database viewing functions). A fourth actor, the AI Engine, represents the external OpenAI GPT service that responds to chatbot queries.

3.4.1 Use Case: UC-01 — First Visit and Consent Decision

Actor: Anonymous Visitor. Precondition: User has no existing consent preference stored in localStorage. Main Flow: (1) Visitor loads any page of the IMR International website. (2) System renders the requested page content fully. (3) After a 1.2 second delay, the consent banner slides into view from the bottom of the screen. (4) Visitor reads the consent explanation text. (5a) Visitor clicks "Accept Analytics" — system stores "accepted" in localStorage, logs consent event to server with timestamp, initializes session UUID via crypto.randomUUID(), and begins tracking page view events. (5b) Visitor clicks "Decline" — system stores "rejected" in localStorage, logs refusal to server, and takes no further tracking action. Post-condition: Consent decision is stored persistently. Banner will not appear on subsequent visits from the same browser.

3.4.2 Use Case: UC-02 — Behavioral Tracking Session

Actor: Consenting Visitor. Precondition: Visitor has previously accepted consent. Main Flow: (1) Visitor loads any page; system detects accepted consent in localStorage. (2) Session UUID is generated if not already exists for this tab. (3) SHA-256 hash of UUID is computed client-side using SubtleCrypto API. (4) Page view event (hash, page path, timestamp) is transmitted to /api/events endpoint. (5) Visitor navigates to additional pages — steps 3-4 repeat for each navigation. (6) Visitor clicks tracked interactive elements — click events are transmitted. (7) On browser tab close, session-end event with duration is transmitted via navigator.sendBeacon(). Post-condition: Complete session behavior is recorded in analytics_events table with no PII.

3.4.3 Use Case: UC-03 — Admin Analytics Access

Actor: Administrator. Precondition: Administrator knows credentials. Main Flow: (1) Administrator navigates to /admin/login. (2) System displays secure login form. (3) Administrator enters username and password. (4) System validates credentials against configured values. (5) On success, server creates authenticated session, sets HttpOnly session cookie. (6) Browser redirects to /admin/analytics. (7) Dashboard loads, fetching aggregate data from /api/analytics/summary, /api/analytics/daily, /api/analytics/weekly endpoints. (8) Administrator selects time range (Daily/Weekly/Monthly) — dashboard refreshes charts. (9) Administrator clicks "Export CSV" to download current data snapshot. (10) Administrator navigates to database viewer at /admin/database to browse raw records. (11) Administrator clicks "Sign Out" — server destroys session.

3.4.4 Use Case: UC-04 — Admission Enquiry Submission

Actor: Anonymous Visitor or Consenting Visitor. Precondition: Visitor is on the /admission/enquiry page. Main Flow: (1) Visitor fills in the enquiry form with full name, email address, phone number, program of interest, and optional message. (2) Client-side validation checks for required fields and email format. (3) Visitor submits the form. (4) System transmits the form data to /api/enquiries endpoint via HTTP POST. (5) Server validates and sanitizes input server-side. (6) Server inserts record into enquiries table with timestamp. (7) Server returns success response. (8) Client displays confirmation message to visitor. Post-condition: Enquiry record permanently stored in database for admissions team review.

3.4.5 Use Case: UC-05 — Chatbot Interaction

Actor: Visitor (any consent status). Precondition: Visitor is on any page of the website. Main Flow: (1) Visitor clicks the green chat bubble in the bottom-right corner. (2) Chat interface expands with a welcome message. (3) Visitor types a question about IMR International (e.g., "What programs does IMR offer?"). (4) Client transmits the conversation history to /api/chat endpoint. (5) Server forwards the conversation to OpenAI GPT API with a system prompt containing IMR International's institutional information. (6) GPT generates a contextually relevant response. (7) Response is returned to client and displayed in the chat interface. (8) Conversation history is appended to the messages table for admin review. Post-condition: Visitor receives accurate, contextual response; conversation stored for analysis.

4.1 System Architecture

The system follows a three-tier architecture, separating presentation, application logic, and data storage into distinct, independently deployable layers. This architectural pattern promotes maintainability by isolating concerns — changes to the user interface do not require modifications to business logic, and database schema changes can be implemented without frontend changes provided the API contract is maintained. The three tiers communicate exclusively through well-defined interfaces: the presentation tier communicates with the application tier through a RESTful HTTP API, and the application tier communicates with the data tier through parameterized SQL queries via the Drizzle ORM.

4.1.1 Presentation Tier (Frontend)

The presentation tier is a React-based Single Page Application (SPA) served by a Vite development server in development mode and compiled into optimized static assets for production deployment. The SPA architecture enables fluid, transition-rich navigation between pages without full browser reloads, which is important both for user experience quality and for the accuracy of session-level behavioral tracking (SPA navigation events are virtual and require explicit tracking as opposed to HTTP-level page loads).

The frontend is organized into a modular component hierarchy. Top-level layout components (TopBar, Navigation, Footer) are shared across all pages. Page-level components (home page, program detail pages, admin dashboard, etc.) compose these shared components with page-specific content. Sub-components encapsulate discrete UI units (ConsentBanner, Chatbot, StatCard, ChartContainer) that can be developed, tested, and modified independently. State management is handled through React's native useState and useContext hooks, augmented by React Query for server state (API data fetching, caching, and synchronization).

Client-side routing is managed by Wouter, a lightweight routing library that intercepts browser navigation events and renders the appropriate page component based on the current URL path. The routing configuration maps URL patterns (e.g., /:category/:slug for dynamic page routes) to page components, enabling the single-page structure while maintaining URL addressability for all content pages.

4.1.2 Application Tier (Backend API)

The application tier is an Express.js HTTP server that exposes a RESTful API consumed by the frontend. All API endpoints are prefixed with /api/ to distinguish them from frontend routes. The server is responsible for business logic enforcement (consent validation before accepting tracking events, authentication checking before serving admin data), data transformation (computing aggregates, formatting responses), and database interaction (query construction, result parsing).

The server is organized into a modular route structure, with separate route handler modules for analytics, consent, events, authentication, admin, enquiry, and chat functions. A shared middleware layer handles cross-cutting concerns including CORS configuration, JSON body parsing, session management, and the visitor logging function that intercepts all incoming requests.

In production, the Express server additionally serves the compiled frontend static assets, functioning as both the API server and the web server for the SPA. This unified serving model simplifies deployment configuration by requiring only a single process and single port allocation.

4.1.3 Data Tier (PostgreSQL Database)

The data tier is a PostgreSQL 16 relational database maintained by the Replit cloud platform. The database stores all persistent application data including behavioral events, consent records, visitor logs, user accounts, enquiry submissions, and chatbot conversations. All database interactions from the application tier use parameterized queries through the Drizzle ORM, preventing SQL injection attacks.

The database connection is managed through a connection pool (the pg library's Pool class) that maintains a configurable number of concurrent database connections, reusing connections across requests to minimize connection establishment overhead. The express-session middleware uses a PostgreSQL-backed session store (connect-pg-simple) that stores session data in a dedicated sessions table, enabling session persistence across server restarts.

4.4 Security Architecture

Security is embedded throughout the architecture rather than treated as an afterthought. The following security mechanisms are implemented at multiple layers of the system.

4.4.1 Transport Security

All traffic between clients and the production server is encrypted using TLS 1.2 or higher, enforced by the Replit hosting platform's HTTPS termination layer. HTTP Strict Transport Security (HSTS) headers prevent downgrade attacks by instructing browsers to only connect via HTTPS for subsequent visits. The application does not store or log any plaintext sensitive data in server logs.

4.4.2 Session Security

Admin sessions are managed by express-session with a PostgreSQL session store. Session cookies are configured with the HttpOnly flag (preventing JavaScript access, mitigating XSS session theft) and the Secure flag in production (ensuring transmission only over HTTPS). The SameSite attribute is set to Lax, providing protection against cross-site request forgery (CSRF) attacks on state-changing operations. Session secrets are drawn from environment variables, never hardcoded in source code.

4.4.3 Input Validation and SQL Injection Prevention

All API endpoints validate incoming request bodies for required fields and appropriate data types before processing. Database interactions use parameterized queries exclusively through the Drizzle ORM's query builder and the pg library's native parameterized query support — user-supplied values are never concatenated into SQL strings, preventing SQL injection attacks. The admin database viewer additionally validates the requested table name against an allowlist of permitted tables before executing any query.

4.4.4 Cryptographic Anonymization

The cryptographic anonymization pipeline is the most distinctive security feature of this system. Client-side session UUIDs are generated using crypto.randomUUID(), which uses the browser's CSPRNG and produces a 128-bit random value with negligible collision probability. Before any transmission to the server, this UUID is processed through SHA-256 using the browser's SubtleCrypto API. The resulting 256-bit hash is then hex-encoded and transmitted as the session identifier. The server stores only this hash, never the original UUID. Since SHA-256 is a cryptographic one-way function, recovering the original UUID from the stored hash requires computational effort equivalent to finding a preimage for SHA-256, which is beyond the capability of any known adversary with current technology.

4.4.5 Admin Access Control

All analytics data, database viewer functionality, and administrative operations are protected by a dedicated admin authentication layer separate from the user authentication system. The requireAdmin middleware function checks for the presence of a valid admin session flag before allowing access to any admin-prefixed API endpoint. Unauthenticated requests to admin endpoints receive a 401 Unauthorized response without any information disclosure. The admin database viewer further enforces a whitelist of accessible tables — requests for any table not on the whitelist receive a 403 Forbidden response, preventing access to internal system tables (pg_catalog, information_schema) even by authenticated administrators.

4.6 Data Flow Design

The following describes the complete data flow for the primary use case — a new visitor arriving, granting consent, browsing multiple pages, and the resulting data appearing in the admin dashboard.

Step 1 — Initial Page Load: Visitor's browser requests the root URL. The Replit hosting layer terminates TLS and proxies the request to the Express server. In production, Express serves the compiled React SPA index.html. The browser downloads and parses the JavaScript bundle, React renders the home page component, and the Vite/React hydration process completes.

Step 2 — Visitor Log Entry: Simultaneously with the page render, the useTracker hook's useEffect fires, calling trackPageView("/"). The logVisit function uses navigator.sendBeacon() to fire a POST request to /api/visitor-log with the current page path and document.referrer. The Express visitor logging middleware extracts the raw IP from the X-Forwarded-For header, computes SHA-256 and takes the first 16 hex characters, extracts the Accept-Language header as a country hint, and inserts a record into the visitor_logs table. This entire operation is asynchronous and non-blocking.

Step 3 — Consent Banner Appearance: After 1200ms, the ConsentBanner component's useState timer triggers, setting the visible state to true. Framer Motion animates the banner into view from below.

Step 4 — User Accepts Consent: User clicks "Accept Analytics". The saveConsent("accepted") function is called. localStorage.setItem("imr_consent", "accepted") stores the decision. A POST to /api/consent transmits the status, the server inserts a record into consent_logs with the timestamp and truncated IP. The banner animates out via AnimatePresence.

Step 5 — Tracking Begins: The useTracker hook re-evaluates hasConsent(), which now returns true. The pending page view for the current page is sent to /api/events with the hashed session ID, event type "page_view", and page path "/".

Step 6 — Navigation: User clicks a navigation link. Wouter intercepts the click, updates the URL without a full page reload, and renders the new page component. useTracker's useEffect fires again, calling trackPageView(newPath), which calls sendEvent("navigation", newPath) and then sendEvent("page_view", newPath), creating two events in analytics_events.

Step 7 — Admin Dashboard View: Administrator logs in, navigates to /admin/analytics. React Query fetches /api/analytics/summary, /api/analytics/daily, /api/analytics/interests. Express executes GROUP BY queries on analytics_events and visitor_logs. Results are returned as JSON. Recharts renders the data as LineChart, BarChart, and PieChart components. The admin sees real-time aggregate data from the current visitor's session within seconds.

5.4 Tracking and Analytics Implementation

The tracking system is implemented in the tracker.ts module, which exports a clean public API consumed by the useTracker.ts custom hook. The separation between the utility module (which contains all logic) and the hook (which integrates with React's lifecycle) enables the utility to be used in non-component contexts if needed.

// artifacts/imr-international/src/lib/tracker.ts (complete)
const BASE_URL = import.meta.env.BASE_URL?.replace(/\/$/, "") || "";
const API_BASE = BASE_URL;

// Session ID — lives only for this browser tab
let sessionId: string | null = null;
let sessionStart: number = Date.now();

function getSessionId(): string {
  if (!sessionId) {
    sessionId = crypto.randomUUID(); // CSPRNG-based UUID v4
    sessionStart = Date.now();
  }
  return sessionId;
}

function hasConsent(): boolean {
  return localStorage.getItem("imr_consent") === "accepted";
}

// SHA-256 hash using SubtleCrypto Web API
async function hashSessionId(id: string): Promise<string> {
  const encoder = new TextEncoder();
  const data = encoder.encode(id);
  const hashBuffer = await crypto.subtle.digest("SHA-256", data);
  const hashArray = Array.from(new Uint8Array(hashBuffer));
  return hashArray.map(b => b.toString(16).padStart(2, "0")).join("");
}

async function sendEvent(eventType: string, page: string, element?: string) {
  if (!hasConsent()) return; // Gate: only track with consent
  try {
    const rawSessionId = getSessionId();
    const hashedId = await hashSessionId(rawSessionId); // Hash before sending
    await fetch(`${API_BASE}/api/events`, {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({
        sessionId: hashedId,   // Only the hash, never the raw UUID
        eventType,
        page,
        element: element ?? null,
      }),
    });
  } catch {
    // Silent fail — analytics must never break user experience
  }
}

// Server-side visitor log (no consent required — no PII stored)
function logVisit(path: string) {
  try {
    navigator.sendBeacon(
      `${API_BASE}/api/visitor-log`,
      new Blob(
        [JSON.stringify({ page: path, referrer: document.referrer || null })],
        { type: "application/json" }
      )
    );
  } catch {
    fetch(`${API_BASE}/api/visitor-log`, {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ page: path, referrer: document.referrer || null }),
      keepalive: true,
    }).catch(() => {});
  }
}

export function trackPageView(path: string) {
  logVisit(path);            // Always — no consent needed, no PII
  sendEvent("page_view", path); // Only when consent given
}

export function trackClick(element: string, page: string) {
  sendEvent("click", page, element);
}

export function trackNavigation(to: string) {
  sendEvent("navigation", to);
}

export function trackSessionEnd() {
  if (!hasConsent() || !sessionId) return;
  const durationSeconds = Math.round((Date.now() - sessionStart) / 1000);
  sendEvent("session_end", window.location.pathname, 
            `duration:${durationSeconds}s`);
}

export async function saveConsent(status: "accepted" | "rejected" | "withdrawn") {
  localStorage.setItem("imr_consent", status);
  try {
    await fetch(`${API_BASE}/api/consent`, {
      method: "POST",
      headers: { "Content-Type": "application/json" },

}

export function getConsent(): string | null {
  return localStorage.getItem("imr_consent");
}

Several important implementation decisions merit detailed explanation. The hashSessionId function uses the SubtleCrypto API, which is available natively in all modern browsers and performs the SHA-256 computation asynchronously in the browser's native cryptographic engine. Using native SubtleCrypto rather than a JavaScript SHA-256 library provides two advantages: it avoids an additional bundle dependency, and it uses hardware-accelerated cryptography where available, making the hashing operation essentially instantaneous.

The use of navigator.sendBeacon() for the visitor log is deliberate and significant. sendBeacon() is designed for exactly this use case — transmitting small payloads to a server without blocking or delaying page navigation. Unlike fetch(), which can be cancelled if the page navigates away before the request completes, sendBeacon() is guaranteed to complete even if the page unloads during the transmission. This reliability is particularly important for session-end events triggered on the unload event, where the page is closing and any fetch() requests would typically be cancelled.

The useTracker hook integrates the tracker with React's lifecycle through useEffect, automatically tracking page views on route changes and registering a window unload handler for session-end events:

// artifacts/imr-international/src/lib/useTracker.ts
import { useEffect } from "react";
import { useLocation } from "wouter";
import { trackPageView, trackSessionEnd } from "./tracker";

export function useTracker() {
  const [location] = useLocation();

  useEffect(() => {
    trackPageView(location);
  }, [location]); // Re-fires on every route change

  useEffect(() => {
    const handleUnload = () => trackSessionEnd();
    window.addEventListener("beforeunload", handleUnload);
    return () => window.removeEventListener("beforeunload", handleUnload);
  }, []);
}

// artifacts/api-server/src/routes/events.ts (excerpt)
import { Router } from "express";
import { pool } from "@workspace/db";

const router = Router();

router.post("/", async (req, res) => {
  const { sessionId, eventType, page, element } = req.body;

  // Validate required fields
  if (!sessionId || !eventType || !page) {
    return res.status(400).json({ 
      error: "sessionId, eventType, and page are required" 
    });
  }

  // Validate event type against allowlist
  const VALID_EVENTS = ["page_view","click","navigation","session_end"];
  if (!VALID_EVENTS.includes(eventType)) {
    return res.status(400).json({ error: "Invalid event type" });
  }

  // Verify sessionId looks like a SHA-256 hash (64 hex chars)
  if (!/^[0-9a-f]{64}$/.test(sessionId)) {
    return res.status(400).json({ error: "Invalid session ID format" });
  }

  await pool.query(
    `INSERT INTO analytics_events 
       (hashed_session_id, event_type, page, element) 
     VALUES ($1, $2, $3, $4)`,
    [sessionId, eventType, page, element ?? null]
  );

  return res.json({ success: true });
});

export default router;

The events endpoint includes server-side validation that the submitted sessionId matches the expected format of a SHA-256 hash (exactly 64 lowercase hexadecimal characters). This validation serves as a defense-in-depth measure: even if an attacker attempts to inject a raw UUID or other identifier as the sessionId, the server will reject it with a 400 error rather than storing it. This ensures the privacy property that only hashed identifiers are ever stored in the database, even in adversarial scenarios.

7.3 Privacy Compliance Results

A structured evaluation of the system's privacy compliance was conducted against the key requirements of GDPR and DPDPA. The following compliance findings were established.

Principle of Lawfulness, Fairness, and Transparency (GDPR Art. 5(1)(a)): The system satisfies this principle through the consent banner mechanism, which provides clear and accessible information about the nature and purpose of data collection before any collection begins. The privacy notice on the consent banner describes the anonymization technique (SHA-256 hashing) in accessible language. The admin dashboard's "Privacy badge" component further reinforces the anonymous nature of the collected data.

Principle of Purpose Limitation (GDPR Art. 5(1)(b)): Data is collected exclusively for the specified purpose of understanding website usage patterns to improve institutional services. No mechanism exists within the system to repurpose collected behavioral data for commercial advertising, user profiling, or sharing with third parties. The database structure provides no columns that would support repurposing — there is no user identifier linking analytics events to registered users, preventing the merger of behavioral data with personal account information.

Principle of Data Minimization (GDPR Art. 5(1)(c)): The system collects only the data necessary for its stated analytical purpose: page path, event type, hashed session identifier, and timestamp. IP addresses are reduced to a 16-character hash fragment — far less than what would be needed for any user identification purpose but sufficient for basic network-level deduplication. No demographic data, device identifiers, or browsing history outside the institution's own website is collected.

Consent Requirements (GDPR Art. 7): The consent implementation satisfies all Article 7 requirements: consent is requested before processing begins (not after), the request is clearly distinguishable from other matters (the banner has a single clear purpose), users are informed of their right to withdraw (stated in the banner text), and withdrawal is as easy as granting consent (both options are equal-prominence buttons on the same interface).

Right to be Informed (GDPR Art. 13/14): The consent banner provides the required information: the purpose of processing, the data minimization approach, and the contact for further information (implied through the institutional contact details prominently displayed in the website header). A full privacy notice page is recommended as a future enhancement to fully satisfy the detailed disclosure requirements of Articles 13 and 14.

The testing phase validated all implemented features against the requirements specification through 16 functional test cases, 11 security and privacy test cases, and 11 performance measurements, achieving 100% pass rate.

The most significant technical contribution of this work is the implementation and validation of a complete, end-to-end privacy-preserving behavioral analytics pipeline in which personally identifiable information is cryptographically irreversible at every storage boundary. The UUID generation → SHA-256 hashing → server-side validation → parameterized storage pipeline, implemented across three distinct layers of the technology stack and validated through comprehensive testing, constitutes a robust technical demonstration that meaningful behavioral analytics can be conducted without any possibility of user re-identification.

The system is currently deployed on the Replit cloud platform and operational as the official web presence for IMR International, Bhubaneswar. This production deployment distinguishes this project from many academic implementations — the system is not a prototype or a simulation, but a live institutional resource receiving real visitor traffic and producing actionable analytics data for institutional decision-making.

8.2 Contributions

This project makes the following specific contributions to the fields of privacy-aware software engineering and educational technology:

  • Reference implementation of consent-first analytics: The system provides a complete, documented, open-architecture reference implementation of a web analytics system that enforces consent as an absolute prerequisite for data collection, with no fallback to default tracking. This implementation demonstrates that consent enforcement imposes no significant technical overhead and is fully compatible with rich analytical capability.
  • Client-side SHA-256 anonymization pipeline: The specific pipeline of CSPRNG UUID generation → SubtleCrypto SHA-256 hashing → format validation → parameterized storage, implemented end-to-end and validated through security testing, represents a replicable pattern that other developers can adopt in their own analytics implementations.
  • Two-tier logging architecture: The conceptual innovation of maintaining two parallel data streams — consent-gated detailed behavioral analytics and consent-free minimal server-side visit logs — provides a practical resolution to the tension between operational completeness (needing some basic visit data even without consent) and privacy compliance (not tracking behavior without consent). This architecture is original to this project and not found in existing analytics platforms.
  • Practical GDPR and DPDPA compliance demonstration in Indian educational context: This project provides the first known example of a purpose-built GDPR/DPDPA-compliant web analytics implementation deployed by an educational institution in Odisha, India, demonstrating that compliance with international privacy standards is achievable within the resource constraints typical of Indian higher education institutions.
  • Complete institutional web platform: The IMR International website, comprising 40+ content pages, full navigation, admission enquiry management, user authentication, and AI-powered assistance, provides a demonstration that a complete, professionally developed institutional web presence can be built and deployed through a modern monorepo architecture without specialized DevOps expertise.

8.3 Limitations

The project acknowledges the following limitations that could be addressed in future iterations:

  • Single-administrator credential system: The current implementation supports only a single admin account with credentials configurable through environment variables. For institutions with multiple staff members requiring different levels of administrative access, a role-based admin account system with database-backed credentials and individual user management would be necessary.
  • No consent withdrawal data erasure workflow: While users can withdraw consent by declining the banner on subsequent visits, the system does not automatically delete previously collected behavioral data upon withdrawal. GDPR's right to erasure (Art. 17) may require such a deletion workflow in fully compliant deployments, though the anonymized nature of the collected data means it is debatable whether the stored data constitutes personal data subject to erasure rights.
  • English-only interface: The consent banner, website content, and admin dashboard are implemented in English only. For maximum accessibility to the diverse student population of Odisha and India, Odia language support and other regional language options would significantly improve the consent banner's accessibility to non-English-reading visitors.
  • Limited cross-session insights: Because session identifiers are tab-scoped and not persisted across browser restarts, the system cannot track returning visitors as the same entity across separate browsing sessions. This is a deliberate privacy protection but limits the ability to analyze return visit patterns, long-term engagement, and conversion journeys that span multiple sessions.
  • No bot/crawler filtering: The visitor_logs and analytics_events tables currently receive records from automated crawlers and bots (search engine crawlers, monitoring services, etc.) that inflate visit counts. A user agent filtering system to exclude known bot patterns from analytics aggregations would improve the accuracy of traffic metrics.
  • Static AI system prompt: The AI chatbot's institutional knowledge is embedded in a static system prompt. As IMR International's programs, fees, and facilities change, the system prompt must be manually updated to reflect current information. A content management interface for updating the AI knowledge base without modifying code would improve maintainability.

8.4 Future Enhancements

The following enhancements are proposed for future development phases, building upon the foundation established by this project:

  • Real-time analytics dashboard: Implementation of WebSocket-based real-time data streaming would enable the admin dashboard to display live visitor counts and page views as they occur, rather than requiring manual refresh. The Socket.io library integrates naturally with Express.js and would enable this capability with minimal architectural change.
  • Multi-admin role system: Development of a database-backed multi-user admin system with role-based access control (RBAC) allowing administrators to be assigned one of several roles (Viewer, Analytics, Full Access) would make the system appropriate for organizations with multiple staff members. This would include a user management interface within the admin panel and bcrypt-protected admin passwords stored in a dedicated admin_users table.
  • Automated consent withdrawal and data erasure: Implementation of a self-service "Forget Me" workflow triggered by the user's withdrawal of consent, which would delete all analytics events associated with any hashed session IDs that can be attributed to the withdrawing user's session (without compromising the anonymization of other users' data), would complete the GDPR compliance profile of the system.
  • Funnel analysis and goal tracking: Addition of configurable conversion funnel visualization to the admin dashboard — tracking the progression of visitors through defined sequences of pages (e.g., Home → Programs → Specific Program → Enquiry Form → Submission Confirmation) — would provide actionable insights for optimizing the institution's digital enrollment funnel.
  • A/B testing framework: Integration of a lightweight A/B testing capability that allows randomized assignment of visitors to page variants and measurement of behavioral differences between variants would enable data-driven optimization of content presentation and calls-to-action.
  • Automated report generation and email distribution: Development of a scheduled report generation system that compiles weekly analytics summaries and emails them to configured administrative recipients would reduce the need for administrators to actively log into the dashboard to stay informed about website performance.
  • Regional language support: Internationalization (i18n) of the consent banner and key user-facing interface elements in Odia, Hindi, and other regional languages would significantly improve accessibility and demonstrate cultural sensitivity for the institution's primary audience.
  • Machine learning pattern detection: Integration of anomaly detection algorithms (such as Isolation Forest or Z-score outlier detection) applied to the accumulated behavioral time series data would automatically flag unusual traffic patterns — potential spam bot activity, denial of service attempts, or genuine viral content events — for administrative attention.
  • Heatmap visualization: Implementation of click position tracking (recording normalized X,Y coordinates of click events relative to the viewport) and visualization as heatmap overlays on page screenshots would provide visual UX insights about which page areas attract the most user attention and interaction.
  • Mobile application extension: Development of a companion React Native mobile application version of the website, sharing the same backend API and analytics pipeline, would extend the institution's digital reach to mobile-native users while maintaining the same privacy-by-design analytics approach. The Expo framework, already available in the Replit ecosystem, would enable rapid cross-platform mobile development.

The foundation established by this project is architecturally positioned to support all of these enhancements without requiring fundamental restructuring. The RESTful API contract is extensible through additional endpoints, the React frontend's component architecture supports new UI features, and the PostgreSQL database schema can be expanded through additional tables and columns as new features require new data storage capabilities.

In conclusion, the Consent-Based Anonymous User Behavior Monitoring System, as implemented and documented in this report, successfully demonstrates that privacy-by-design is not merely a theoretical aspiration but a practically achievable engineering discipline that delivers genuine, actionable value to institutional stakeholders while fully respecting the rights and privacy of the individuals whose behavior it measures. The system stands as both a working production deployment serving IMR International and a contribution to the body of knowledge on ethical, consent-aware web analytics architecture.

      ORDER BY ordinal_position
    `, [t]);

    return {

      columns: rows
        .filter(r => !omit.includes(r.column_name))
        .map(r => ({ name: r.column_name, type: r.data_type })),
    };

  const { table } = req.params;
  if (!ALLOWED_TABLES[table]) {
    return res.status(403).json({ error: "Table access not allowed" });
  }
  const page = Math.max(1, parseInt(String(req.query.page ?? "1"), 10));
  const limit = Math.min(50, parseInt(String(req.query.limit ?? "20"), 10));
  const offset = (page - 1) * limit;
  const search = String(req.query.search ?? "").trim();
  const omit = ALLOWED_TABLES[table]?.omit ?? [];
  const { rows: colRows } = await pool.query(`
    SELECT column_name FROM information_schema.columns
    WHERE table_schema='public' AND table_name=$1
    ORDER BY ordinal_position`, [table]);
  const cols = colRows.map(r => r.column_name as string)
    .filter(c => !omit.includes(c));
  const selectCols = cols.map(c => `"${c}"`).join(", ");
  let countQuery = `SELECT COUNT(*) AS count FROM ${table}`;
  let dataQuery = `SELECT ${selectCols} FROM ${table}`;
  const params: unknown[] = [];
  if (search) {
    const conditions = cols.map((c, i) => 
      `CAST("${c}" AS TEXT) ILIKE $${i + 1}`).join(" OR ");
    countQuery += ` WHERE ${conditions}`;
    dataQuery += ` WHERE ${conditions}`;
    params.push(...cols.map(() => `%${search}%`));
  }
  dataQuery += ` ORDER BY 1 DESC LIMIT $${params.length+1} 
                OFFSET $${params.length+2}`;
  const [countResult, dataResult] = await Promise.all([
    pool.query(countQuery, params),
    pool.query(dataQuery, [...params, limit, offset]),
  ]);
  const total = Number(countResult.rows[0]?.count ?? 0);
  return res.json({
    table, columns: cols, rows: dataResult.rows, total,
    page, limit, pages: Math.ceil(total / limit),
  });
});

      pool.query(`SELECT AVG(CAST(REPLACE(element,'duration:','') 
                  AS INTEGER)) AS avg FROM analytics_events 
                  WHERE event_type='session_end' 
                  AND element LIKE 'duration:%'`),
      pool.query(`SELECT page, COUNT(*) AS visits FROM analytics_events 
                  WHERE event_type='page_view' GROUP BY page 
                  ORDER BY visits DESC LIMIT 1`),
    ]);
  res.json({
    totalPageViews: Number(totalEvents.rows[0]?.count ?? 0),
    uniqueSessions: Number(uniqueSessions.rows[0]?.count ?? 0),
    avgSessionDuration: Math.round(
      Number(avgDuration.rows[0]?.avg ?? 0)),
    topPage: topPage.rows[0]?.page ?? "/",
  });
});

router.get("/daily", async (_req, res) => {
  const { rows } = await pool.query(`
    SELECT DATE(timestamp AT TIME ZONE 'Asia/Kolkata') AS date,
      COUNT(*) FILTER (WHERE event_type='page_view') AS page_views,
      COUNT(DISTINCT hashed_session_id) AS unique_sessions
    FROM analytics_events
    WHERE timestamp >= NOW() - INTERVAL '30 days'
    GROUP BY DATE(timestamp AT TIME ZONE 'Asia/Kolkata')
    ORDER BY date ASC
  `);
  res.json(rows.map(r => ({
    date: r.date,
    pageViews: Number(r.page_views),
    sessions: Number(r.unique_sessions),
  })));
});

router.get("/interests", async (_req, res) => {
  const { rows } = await pool.query(`
    SELECT page, COUNT(*) AS visits FROM analytics_events
    WHERE event_type = 'page_view'
    GROUP BY page ORDER BY visits DESC LIMIT 10
  `);
  res.json(rows.map(r => ({
    page: r.page, visits: Number(r.visits)
  })));
});

  name          TEXT NOT NULL,
  email         TEXT NOT NULL UNIQUE,
  phone         TEXT,
  password_hash TEXT NOT NULL,
  created_at    TIMESTAMP NOT NULL DEFAULT NOW()
);

-- 2. Behavioral analytics events (CONSENT-GATED)
-- Only populated when visitor has explicitly accepted consent.
-- hashed_session_id is SHA-256 of client-generated UUID.
-- Raw UUID never transmitted to server.
CREATE TABLE IF NOT EXISTS analytics_events (
  id                SERIAL PRIMARY KEY,
  hashed_session_id TEXT NOT NULL,
  event_type        TEXT NOT NULL 
    CHECK (event_type IN ('page_view','click','navigation','session_end')),
  page              TEXT NOT NULL,
  element           TEXT,
  timestamp         TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 3. Consent decision audit log
-- Records every consent acceptance, rejection, and withdrawal.
-- ip_address stores only partial/truncated IP, never full address.
CREATE TABLE IF NOT EXISTS consent_logs (
  id         SERIAL PRIMARY KEY,
  status     TEXT NOT NULL 
    CHECK (status IN ('accepted','rejected','withdrawn')),
  ip_address TEXT,
  timestamp  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 4. Server-side visitor log (CONSENT-FREE, MINIMAL)
-- Records for all visitors regardless of consent status.
-- ip_hash: first 16 chars of SHA-256(raw_ip). Never raw IP.
-- country_hint: Accept-Language header only. No geolocation.
-- user_agent: Browser string, truncated to 300 chars.
CREATE TABLE IF NOT EXISTS visitor_logs (
  id           SERIAL PRIMARY KEY,
  page         TEXT NOT NULL,
  referrer     TEXT,
  user_agent   TEXT,
  ip_hash      TEXT,
  country_hint TEXT,
  visited_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 5. Admission enquiries
-- Stores enquiry form submissions from prospective students.
-- Contains genuine PII (name, email, phone) submitted
-- voluntarily by the enquirer for the purpose of follow-up.
CREATE TABLE IF NOT EXISTS enquiries (
  id         SERIAL PRIMARY KEY,
  name       TEXT NOT NULL,
  email      TEXT NOT NULL,
  phone      TEXT NOT NULL,
  program    TEXT NOT NULL,
  message    TEXT,
  created_at TIMESTAMP NOT NULL DEFAULT NOW()
);

-- 6. AI Chatbot conversation threads
CREATE TABLE IF NOT EXISTS conversations (
  id         SERIAL PRIMARY KEY,
  title      TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 7. Individual chatbot messages
CREATE TABLE IF NOT EXISTS messages (
  id              SERIAL PRIMARY KEY,
  conversation_id INTEGER NOT NULL REFERENCES conversations(id)
                    ON DELETE CASCADE,
  role            TEXT NOT NULL CHECK (role IN ('user','assistant')),
  content         TEXT NOT NULL,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
